## 时间线（过程 + 细节）

### 04-05 早段 - 从公网访问实验回到本地可验证链路

用户关注：
- 是否需要先用 Render 地址测试
- 如果先测本地，应该如何验证

AI 当时判断：
- 优先保证本地链路可用，再进入公网验收
- `cloudflared tunnel --url http://127.0.0.1:3000` 只能作为临时暴露手段，不应替代真实产品链路验证
- 先确认本地 Web、local bridge、登录态、设备状态四条链是否打通，才能判断公网问题属于应用问题还是隧道噪音

实际暴露的问题：
- `cloudflared` 日志中大量出现：
  - `malformed HTTP response "Unauthorized"`
  - `_next/webpack-hmr` websocket 相关错误
  - 后续还有长时间 `control stream encountered a failure while serving`

用户侧反馈重点：
- 希望先把“本地如何测”说清楚
- 不希望过早被 Render / Tunnel 的部署噪音带偏

结论收敛：
- 本地优先是对的
- Quick Tunnel 日志不是当前最关键的验收信号
- 真正要关注的是产品内部状态是否清晰、错误是否可恢复

### 04-05 中段 - 默认设备离线错误把问题从“技术异常”暴露成“认知混乱”

用户实际遇到的错误：
- 工作区报错：
  - `Your default Relay device is currently offline.`
- 位置：
  - [`src/lib/api/bridge.ts`](/Users/ruska/project/web-cli/src/lib/api/bridge.ts)

AI 当时判断：
- 当前不是单点接口报错问题
- 而是“默认设备、当前本机、历史设备、session 有效性”之间的状态表达不够清楚
- 单纯修一个错误提示，不足以解决用户混乱

用户随后明确提出的优化目标：
- 自动识别当前本机是否已绑定，尽量自动通过
- 设置页把“我的设备列表”和“默认设备”做得更清晰
- 减少 session 过期 / 默认设备离线带来的认知混乱
- 给工作区补更明确的错误态和恢复态

这一步的产品转向：
- 从“功能是否存在”转向“状态表达是否让人一眼看懂”
- 从工程视角的设备模型，转向用户视角的连接结果

### 04-05 下午 - 对 cloudflared 报错的优先级重新排序

用户继续贴出长段 `cloudflared` 日志，并直接追问：
- “为啥感觉总是在报错呢？需要关注么？”

AI 预期：
- 需要把隧道层噪音和应用层故障拆开
- `HMR websocket Unauthorized` 在开发态经常是外层代理兼容性问题，不一定是产品主故障
- 长时间 `control stream encountered a failure while serving` 更像 quick tunnel 自身稳定性问题，不应作为核心产品验收结论

用户真实关心点：
- 哪些报错需要处理
- 哪些可以暂时忽略
- 不希望把时间消耗在低价值噪音上

最终收敛意见：
- 需要关注的不是 Tunnel 本身是否“干净无红字”
- 而是：
  - 本地登录是否稳定
  - 默认设备选择是否正确
  - 工作区是否能自动恢复
  - Render 公网版本是否稳定

### 04-05 下午 - “我只有一台电脑，为什么会有离线设备”暴露历史设备清理缺口

用户直觉反馈：
- “这里只有一个离线的设备，我只有一个电脑才对吧。”

AI 当时判断：
- 这不是用户理解错误
- 更可能是历史绑定记录、旧设备 ID、旧默认设备偏好残留导致
- 现有 UI 没有把“当前在线设备”和“历史离线设备”区分清楚，也没有给出清理出口

用户随即提出明确需求：
- 增加历史离线设备提示 / 清理能力
- 并要求直接帮忙清理

这一步的产品含义：
- 设置页不能只是设备信息面板
- 还必须提供状态治理能力
- 历史设备如果不能清理，就会持续污染默认设备选择与用户信心

### 04-05 晚些时候 - 目标切换到 v0.1.6 的“体验收口版本”

用户确认：
- 可以开始实现 `0.1.6`
- 随后又明确进入“公网 / Render 的稳定验收阶段”

AI 当时判断：
- `v0.1.6` 不该再是零散修补
- 应该是一次围绕“设备体验收口 + 公网验收”组织的版本
- 需要同时覆盖：
  - 设备绑定体验
  - 默认设备逻辑
  - 错误态 / 恢复态
  - Render 验收路径

用户同步提出的交付需求：
- 给出 Render 验收 checklist
- 明确是否需要推送到 GitHub 完成部署

最终共识：
- 如果 Render 使用 GitHub 集成部署，则需要推送到 GitHub 才能触发对应部署
- 本地验证、公网预览、Render 正式验收，应被定义为不同层级的验收，而不是混成一个动作

### 04-05 晚段 - 用户把目标提升为“Web 登录后自动接力本地 Codex”

用户提出更高层产品愿景：
- 在 Web 版本登录账号后，自动绑定本地设备
- 之后移动端可以直接使用这个产品
- 用户只要本地电脑开着，就可以直接接力使用本地 Codex
- 尽量减少配置步骤

AI 当时预期：
- 这是正确方向
- 但距离该目标仍至少缺几块：
  - 自动发现“当前本机就是可绑定目标”的规则
  - 登录后自动绑定 / 自动设默认设备的策略
  - 移动端对“默认设备在线并可用”的简化入口
  - session 失效 / 设备离线 / 历史残留时的恢复链路

用户侧强调的价值排序：
- 不想让普通用户理解太多“设备管理”概念
- 希望产品看起来像“登录后就能用”
- 配置越少越好

这一步完成的产品收敛：
- 移动端目标不是“管理设备”
- 而是“如果桌面端在线，移动端直接使用”

### 04-05 深夜 - 设置页被重新定义为“连接状态页”，而不是后台管理页

用户继续追问：
- 从设置页面的整个逻辑来说，有哪些优化点
- 是否只需要“登录 / 设备”一个 tab 和“外观”一个 tab
- 希望做整合、简化、优化

AI 当时判断：
- 设置页不应继续沿着“后台控制面板”式复杂结构扩张
- 更适合收敛为两个主视图：
  - 连接与设备
  - 外观与偏好

用户隐含但反复强调的判断标准：
- 页面需要更少概念
- 要更像消费级连接状态页
- 要让“当前是否可用、如果不可用如何恢复”一眼就懂

由此收敛出的设置页方向：
- 弱化“我的设备列表”里的后台管理味道
- 强化：
  - 当前设备
  - 默认设备
  - 在线状态
  - 历史离线清理
  - 恢复操作

### 04-06 凌晨后段 - 用户指出“按阶段拆分”不利于并行推进，要求改为模块化

用户关键修正意见：
- 之前把执行计划分成多个阶段
- 但阶段之间天然有先后依赖，不利于同时推进
- 更希望按相互解耦的模块拆分，而不是按阶段拆分

AI 当时判断：
- 这个修正是成立的
- ADSF 适合作为外层治理框架
- 但内层执行确实更适合模块化组织

最终收敛出的规范：
- `ADSF` 作为外层治理流程
- `Module` 作为内层实现组织方式
- 每个模块都必须写清：
  - 目标
  - 边界
  - 输入 / 输出
  - 显式依赖
  - Done Definition
- 模块之间要尽量解耦，并支持并行推进与独立验证

这一步是本轮最重要的方法论修正之一：
- 用户不只是要一份计划
- 而是要求未来所有计划都具备更强的执行性与并行友好性

## 本轮 AI 预期的演变

早期预期：
- 先打通链路，再看公网
- 先解决错误，再继续加功能

中期预期：
- 设备绑定问题本质上是状态表达问题
- 需要把默认设备、历史设备、session 与恢复路径一起考虑

后期预期：
- 用户真正要的不是“管理设备”
- 而是“登录后自动接上自己那台在线电脑”
- 设置页应该围绕连接状态而不是后台配置展开
- 执行计划必须升级为 ADSF + 模块化标准，不再接受松散文档

## 本轮用户意见与侧重点收敛

- 配置负担要继续下降，用户不应理解过多设备管理概念。
- Web 登录后应尽可能自动绑定本机，并自动形成默认使用路径。
- 移动端目标是直接使用桌面端在线能力，而不是先学会管理设备。
- 设置页应大幅简化，优先保留“连接 / 设备”和“外观 / 偏好”两类主结构。
- 历史离线设备必须可见、可解释、可清理，否则会持续制造默认设备混乱。
- 工作区必须提供明确错误态与恢复态，不能只抛一句离线错误。
- 版本执行计划必须：
  - 覆盖全部新增功能点
  - 每项都有单元测试
  - 最终有 E2E 闭环
  - 严格遵守 ADSF
  - 按解耦模块组织，而不是按串行阶段组织
