### 08:34 - 决定先做前端静态原型，不先接后端

用户输入：
- 希望先把前端页面做出来，先看效果。
- API 和后端对接可以后做。

AI 当时的判断：
- 这个顺序非常合理。
- 原因：
  - 当前最大的未知数不是后端，而是产品气质是否对。
  - 先把前端静态壳子做出来，可以快速验证布局、视觉、信息密度、用户感受。
  - UI 风格定住后，再设计 API 会更准确，因为知道页面真正需要显示什么。

AI 当时做的技术选择：
- 使用 `Next.js` 快速起一个前端原型。
- 页面先全部用静态假数据。
- 只做页面结构和视觉，不接任何真实接口。

### 09:35 - 09:45 - 处理中间的类型与构建问题

在左侧结构重构过程中出现的小问题：
- `TypeScript` 对 section item 推断成联合类型，导致 `item.active` 类型错误。
- 同时一度有未使用变量的 lint 警告。

AI 处理方式：
- 明确定义：
  - `type SectionItem = { title: string; method?: string; active?: boolean }`
- 将 `sections` 显式标注为：
  - `{ title: string; items: SectionItem[] }[]`
- 删除不再需要的旧 `sessions` 数据

这个细节对交接有意义，因为：
- 下一位接手者如果继续把左侧提取成组件，需要保留这个类型模型，不要重新回到松散联合类型。
