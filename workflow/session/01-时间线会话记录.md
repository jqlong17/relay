## 时间线（过程 + 决策细节）

### 06:00 - 需求起点与核心问题

用户输入：
- 想做一个 Web 端命令行产品，界面类似参考图，运行非常丝滑。

AI 观察与建议：
- 首先把问题从“终端皮肤”转为“交互节奏是否像终端”。
- 丝滑的本质在输入、流式反馈、状态切换，而不是黑底样式。

候选方向（当时同时存在）：
- A. 终端风格文档站
- B. 轻量终端模拟器
- C. Agent 工作台（对话 + 执行 + 文件）

取舍理由：
- 仅做终端皮肤难以满足“真的执行任务”与“普通用户”目标。
- 需要同时承载自然语言、执行过程、文件上下文。

阶段结果：
- 讨论进入“Agent 工作台”方向，不再把产品理解为纯终端外观。

### 06:03 - 明确用户画像与交互结构

用户输入：
- 目标用户：普通产品用户。
- 需要真实执行任务，类似 `Claude Code / Codex`。
- 输入：自然语言优先 + 命令模式可切换。
- 布局：左 session / 中对话 / 右文件树或文件内容。
- 丝滑标准：打字不卡、动画自然、流式响应快。

AI 建议：
- 把产品视为“AI coding 工作台”，不是聊天或文档工具。
- 交互应包含“对话 + 执行时间线”的混排。

候选布局：
- A. 纯聊天流 + 右侧文件预览
- B. 聊天流 + 执行块混排 + 右侧文件树/变更

取舍理由：
- 仅聊天会掩盖执行细节，降低信任感。
- 执行块让用户知道“Agent 正在做什么”。

阶段结果：
- 确认三栏布局 + 中间混排执行块。

### 06:07 - 界面结构进一步收敛

AI 建议（补充细节）：
- 左侧：session 历史
- 中间：对话 + 执行过程（read/write/run/diff）
- 右侧：文件树 + 文件内容 + diff 切换

为什么强调执行块：
- Agent 产品核心是“可审计的执行过程”。
- 只有对话会让用户不确定系统是否真的操作了文件。

阶段结果：
- 中间区明确为“对话 + 执行时间线”。

### 06:11 - 技术诉求被明确

用户输入：
- 先做本机工作区。
- 未来希望手机 Web 也能操作本地 Codex。
- 右侧默认文件树，但也要看变更。
- 重点用户场景是 AI coding。

AI 提示的关键约束：
- 必须支持本地文件/命令真实执行。
- 未来跨设备访问是核心方向，架构需要预留。

阶段结果：
- 讨论转向本地优先架构，不走纯浏览器权限方案。

### 06:14 - 技术路线对比与取舍

AI 给出的候选方案：
- A. 纯浏览器方案（File System Access API）
- B. 云端工作区方案（远程执行）
- C. 本地 Agent + Web 工作台（本地 bridge）

AI 推荐方案：
- 选择 C：本地 Agent/bridge + Web 前端。

推荐理由：
- A 方案：权限不稳定、能力受限、难支持复杂命令和长任务。
- B 方案：离用户本机工作区太远，迁移成本高。
- C 方案：真实本地能力 + 未来可通过网络暴露给手机/远程。

阶段结果：
- 确认技术方向：`本地 Agent/bridge + Web 工作台`。

### 06:18 - Codex 源码接入方式讨论

用户输入：
- 本地已有 Codex 源码：`/Users/ruska/开源项目/codex`
- 询问是否复制到当前项目或重新拉取。

AI 执行动作：
- 查看当前项目为空目录。
- 检查本地 Codex 仓库结构与文档。
- 发现 `codex app-server` 提供面向富 UI 的 JSON-RPC 能力。

AI 建议：
- 不复制 monorepo 到当前项目。
- 把 Codex 当外部内核能力，通过 `app-server` 对接。

候选方案：
- A. 直接复制整个源码进 `web-cli`
- B. git submodule / subtree
- C. 独立仓库 + 运行时依赖（推荐）

取舍理由：
- A 维护成本高、更新困难。
- B 仍带来合并负担。
- C 边界清晰、可随时升级上游。

阶段结果：
- 选择方案 C，不复制源码。

### 06:24 - 最终候选：Roam vs Relay

候选比较：
- `Roam`：自由、轻盈，偏“人可漫游”
- `Relay`：连接、中继、接力、持续性，偏“任务连续接续”

AI 推荐：
- 选择 `Relay` 更贴“本地 Agent + Web 控制 + 跨设备接力”。

阶段结果：
- 产品名确认：`Relay`。
