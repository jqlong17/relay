### 14:20 - 用户提出把前端样式抽象成配置，而不是继续散落在代码里

用户输入：
- 希望后续用户可以通过自然语言和大模型交互，去修改某个配置文件，从而改变前端样式。
- 明确表示“唯一真相”最好是配置文件，而不是直接改前端代码。
- 重点强调性能和流畅度优先，不希望为了可配置性牺牲运行手感。

AI 当时的判断：
- 直接把大量前端实现细节暴露给最终用户并不合适，复杂度高，也难以控制结果质量。
- 更合理的方案是：
  - 前端内部保留稳定实现
  - 对外暴露有限、可解释、可验证的配置层
  - 让大模型只改配置文件，而不是改 JSX / CSS 结构

技术建议：
- 配置项优先收敛到真正高价值的全局参数：
  - 字号
  - spacing
  - layout 宽度
  - 若干 surface / color token
- 不建议一开始做“所有 UI 都能自然语言改”的全开放系统。
- 性能层面无需为了这类配置系统直接上 Rust。当前 web 端瓶颈主要不在配置解析，而在：
  - React 渲染
  - 消息流更新
  - 文件树渲染
  - runtime 流式事件处理

阶段结果：
- 产品方向明确为“配置驱动的 UI 定制”，但仅暴露有限 token，不暴露底层实现。

## 下一位接手者建议从这里继续

### 优先级 1：开始定义真实 API，而不是继续做纯视觉

理由：
- 页面结构已经足够明确。
- 如果再继续微调 UI，收益会快速下降。
- 当前最大的产品风险已经不是“长什么样”，而是“真实数据与真实执行怎么进来”。

建议先定义 4 组核心接口：

1. `workspace` 接口
   - 获取 workspace 列表
   - 获取当前 workspace 的 branch / runtime 状态
   - 获取当前 workspace 下的 session 列表

2. `session` 接口
   - 获取 session 列表
   - 获取 session 详情
   - 获取 session 多轮消息
   - 新建 session / 切换 active session

3. `runtime` / `agent` 接口
   - 提交用户输入
   - 启动一次任务执行
   - 流式返回消息、工具调用、状态更新
   - 中断 / 继续

4. `memory` 接口
   - 获取 memory 日历数据
   - 获取某日 memory 列表
   - 基于多个 session 生成 memory 草稿
   - 保存 memory
   - 获取 / 更新 memory automation 规则

### 优先级 2：先写前端 mock contract，再接真实后端

更具体建议：
- 先在前端定义类型和 mock adapter，而不是直接连 Codex runtime。
- 这样可以先把页面状态流走通，减少后端耦合。

推荐新增的文件方向：
- `apps/web/src/types/workspace.ts`
- `apps/web/src/types/session.ts`
- `apps/web/src/types/runtime.ts`
- `apps/web/src/types/memory.ts`
- `apps/web/src/lib/mock-data/*.ts`
- `apps/web/src/lib/api/*.ts`

### 优先级 3：先接“只读接口”，再接“执行接口”

原因：
- 只读接口更容易稳定页面状态。
- 执行接口会牵涉流式传输、任务中断、工具事件、多种 message item 类型，更复杂。

建议顺序：
1. workspace / sessions / memories 的只读数据
2. active session 切换
3. 输入框提交
4. runtime 流式事件
5. memory 保存与 automation
