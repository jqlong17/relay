## 本轮形成的核心经验

这次沉淀下来的，不是某个浏览器特例 hack，而是一个以后都适用的方法论：

### 1. 不要只记“我改了什么”，要记“最终布局到底有没有生效”

移动端键盘问题里，最容易误判的两件事是：
- `focus` 成功了，以为问题已经解决
- `setProperty` 打印出来了，以为布局已经生效

这次明确证明：
- 事件触发不等于布局生效
- 变量更新不等于正确宿主消费到了变量

### 2. 真机视频和结构化诊断要配合，而不是互相替代

真机视频的价值：
- 帮我们判断是不是方向猜错了

结构化诊断工具的价值：
- 帮我们确认问题断在链路的哪一层

E2E 的价值：
- 固化已经确认的回归点，防止以后退回去

### 3. 以后排查类似问题，优先打开诊断工具，而不是先写临时日志

推荐固定动作：
- 先开 `/mobile?debugMobileLayout=1`
- 真机复现一次
- 导出 JSON
- 再根据事件时间线判断问题是断在：
  - event
  - viewport
  - vars
  - computed
  - rect

## 下一步建议

如果下一轮继续围绕真机移动端排查，建议直接按这套流程走：

1. 在真机打开 `/mobile?debugMobileLayout=1`
2. 复现一次问题
3. 复制面板 JSON
4. 先对照 `viewport.update` 与 `layout.sync`
5. 再决定是修 iOS 首帧补偿、宿主变量、还是布局消费公式

这样做的意义是：
- 不再重复回到“先猜是不是浏览器事件丢了”
- 把每次排查都沉淀成能复查、能复用、能对比的结构化证据
