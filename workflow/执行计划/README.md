# workflow/执行计划 目录规范

本目录用于存放 Relay 的版本执行计划（Execution Plans, EP）。

从现在开始，**未来所有新增执行计划都必须严格按照 ADSF 标准撰写**，不再接受只列功能点、只列 TODO、没有 Traceability / Runtime / Finalize 的松散计划。

---

## 1. 强制要求

以后本目录下的每一份新执行计划，都必须满足以下要求：

1. 必须采用 ADSF 三阶段结构
- `A+D — Architect & Draft`
- `S — Synthesize`
- `F — Finalize`

2. 必须包含执行元信息
- EP 编号
- 目标版本
- 当前阶段
- Scope
- 执行口令 / 人类确认状态

3. 必须包含 Traceability Check
- 明确本轮改动影响哪些页面
- 明确本轮改动影响哪些 API
- 明确本轮改动影响哪些核心链路
- 明确本轮改动影响哪些部署 / runtime 环节

4. 必须包含 Runtime 声明
- 是否涉及数据库迁移
- 是否涉及环境变量变更
- 是否涉及构建 / 镜像 / 启动方式变更
- 是否涉及外部依赖与部署配置

5. 必须包含风险点与回滚原则
- 说明主要风险
- 说明出现问题时的回滚边界

6. 必须按模块（Module）拆分
- 执行计划的实现部分必须以**解耦模块**而不是串行阶段来组织
- ADSF 三阶段只作为外层治理框架，不作为内层功能拆分方式
- 每个模块都要有：
  - 目标
  - 边界
  - 输入 / 输出
  - 显式依赖
  - 完成定义（Done Definition）

7. 每个新增功能点都必须配套单元测试要求
- 不允许只写实现，不写测试
- 不允许“后面再补测试”

8. 必须包含最终 E2E 验收
- 至少定义最终闭环场景
- 必须明确本地 / 公网 / 部署后的最终验收方式

9. 必须包含 Finalize 收尾清单
- 文档同步
- Runtime 收尾
- 人工验收
- 输出变更摘要 / 风险 / follow-ups

---

## 2. 统一模板要求

未来执行计划至少应包含以下章节：

1. `0. ADSF 执行元信息`
2. `目标`
3. `本轮验收标准`
4. `设计原则`
5. `模块图（Module Map）`
6. `模块定义`
7. `TDD 拆分`
8. `Finalize 收尾清单`
9. `交付后的用户可见变化`
10. `本轮明确不做`
11. `版本建议`
12. `当前产品决策`

如果缺少上述核心章节，视为该执行计划**不符合规范**。

---

## 3. 测试规范

以后执行计划中的测试要求必须明确分成两层：

### 3.1 单元测试

每个新增功能点都必须写明：

- 对应测试文件
- 需要覆盖的关键场景
- Done 标准
- 所属模块

### 3.2 最终 E2E 测试

每个版本执行计划都必须包含最终闭环测试，至少覆盖：

1. 用户主路径
2. 关键异常路径
3. 恢复路径
4. 部署后真实访问路径

没有 E2E 闭环定义的执行计划，不视为完整计划。

## 5. 禁止事项

以后禁止提交以下类型的执行计划：

- 只有功能清单，没有架构与影响半径分析
- 只有实现步骤，没有测试要求
- 只有单元测试，没有最终 E2E 验收
- 没有 Runtime 声明
- 没有 Finalize 收尾清单
- 没有明确“本轮不做”的范围边界

这类文档一律视为：

- 草稿
- 讨论提纲
- 非正式计划

而不是可执行版本计划。

---

## 6. 执行原则

Relay 后续版本执行计划统一遵循以下原则：

- 文档先行：先计划，再实现
- Traceability 先行：先看影响半径，再动代码
- 测试同步：实现与测试一起生成
- Finalize 强制：没有收尾清单就不算完成
- 用户结果优先：计划必须最终映射到用户可见价值，而不是只描述工程动作
- ADSF 外层治理：使用 `A+D → S → F` 管理计划、执行、验收
- 模块化内层执行：具体实现必须拆成可解耦推进的模块，而不是默认串行阶段

---

## 7. 模块化要求

未来执行计划中的“实现拆分”必须满足以下模块化要求：

1. 模块优先于阶段
- 不再使用“阶段 A/B/C/D”作为主要功能拆分方式
- 可以保留 ADSF 外层阶段，但内层必须是模块

2. 模块必须显式写依赖
- 哪些模块可以独立推进
- 哪些模块依赖其他模块先稳定
- 哪些模块可以并行验证

3. 模块必须能独立验证
- 每个模块都要有自己的单元测试
- 每个模块都要有自己的 Done Definition

4. 模块必须能映射到用户结果
- 模块不是纯技术分层切片
- 模块名称和内容应尽量对应用户感知能力

---

## 8. 结论

**从本 README 生效起，本目录内未来新增的所有执行计划，必须严格按照 ADSF 标准撰写。**

并且：

- **ADSF 用于治理流程**
- **模块化用于组织实现**

如果后续需要新增模板，应以 ADSF 为唯一基线，而不是退回到传统的松散 TODO 文档形式。
