# Relay v0.2.0 产品目标

## 1. 文档目的

本文件用于定义 Relay `v0.2.0` 的产品目标。

`v0.2.0` 不是在 `v0.1.0` 上简单增加几个功能，而是产品能力的一次明显升级：

- 从“本机 Web 工作台”升级为“可远程访问的个人 Agent 工作台”
- 从“只在电脑本地使用”升级为“手机可随时接入自己电脑上的 Agent”

本版本重点不是做大而全的平台，而是把“人在外面，也能稳定地用手机接入家里或办公室电脑上的 Codex 能力”这件事真正做成。

---

## 2. 版本定位

Relay `v0.2.0` 的定位是：

一个支持账号登录、设备在线状态管理、云端中继连接和手机 Web 访问的远程 Agent 工作台。

用户可以在任何地方，通过手机访问自己电脑上常驻运行的 `Relay local bridge`，并通过对话方式调用底层 `codex cli` 持续执行 AI coding 任务。

这个版本的核心不是“多端看数据”，而是：

- 远程可达
- 持续在线
- 对话驱动执行
- 会话同步
- 稳定、快速、不断线

---

## 3. 核心使用场景

用户的典型场景是：

- 电脑在家里或办公室
- 电脑上运行一个常驻的 `Relay local bridge`
- 用户人在外面，通过手机 Web 登录 Relay
- 用户查看自己的在线电脑
- 进入某个 workspace
- 继续通过对话驱动本地 `codex cli` 执行 AI coding 任务

这个场景下，手机端优先级最高的是：

- 对话
- 任务状态
- 流式返回
- 保持连接

当前版本中，手机端对文件树浏览、session 深度编辑、memory 深度编辑都不是最高优先级。

---

## 4. v0.2.0 核心目标

### 4.1 远程访问目标

必须支持：

- 电脑运行并常驻 `Relay local bridge`
- 电脑主动连接云端中继服务
- 用户在非同一网络环境下，通过手机访问该电脑上的 Relay 能力
- 用户不需要手动配置局域网、端口映射或公网地址

当前技术方向明确为：

- `local relay bridge + 云端中继 + Web 前端（桌面 / 手机）`

这意味着：

- 产品可以使用隧道、反向连接、长连接中继等技术实现
- 但这些技术细节不应暴露给用户
- 用户只感知为“设备在线即可连接，设备离线则不可执行”

### 4.2 手机 Web 目标

必须支持一个体验良好的手机 Web 端。

手机端在 `v0.2.0` 的重点能力是：

- 查看设备在线状态
- 切换 workspace
- 发起和继续对话
- 实时查看 AI / runtime 返回
- 查看任务状态

当前版本中，手机端不要求优先支持：

- 文件树浏览
- 文件深度编辑
- session 复杂管理
- memory 深度编辑

也就是说，手机端首先是：

- 一个远程对话式任务入口

而不是：

- 一套完整复刻桌面端所有信息密度的复杂工作台

### 4.3 会话同步目标

必须支持：

- 同一账号下，手机端和电脑端看到一致的 session
- 用户在手机上继续发起的对话，可以回到电脑上继续查看
- 会话内容完整同步，而不只是标题级别元数据

`v0.2.0` 中，session 同步必须覆盖：

- session 基本信息
- 多轮消息内容
- 关键任务状态

### 4.4 Memory 同步目标

必须支持：

- 完整 memory 内容同步到云端数据库
- 多端登录下查看一致的 memory 数据

当前版本重点是：

- memory 数据一致
- 可以在不同端看到同一批记忆

当前版本不要求：

- 手机端优先做深度 memory 编辑工作流

### 4.5 账号目标

必须支持：

- 账号登录
- 多端登录
- 同一账号下关联多设备

账号体系在 `v0.2.0` 中主要承担：

- 身份识别
- 设备归属
- session / memory 云端同步
- 手机端与本地 bridge 的关联

### 4.6 在线状态目标

必须支持：

- 清楚显示某台电脑是否在线
- 当设备在线时允许进入并执行任务
- 当设备离线时明确提示不可执行

用户已经明确要求：

- 如果电脑离线，应直接显示：
  - `此设备离线，无法执行任务`

当前版本不要求：

- 离线时仍然支持远程执行

---

## 5. v0.2.0 主用户旅程

`v0.2.0` 需要完整支持如下主链路：

1. 用户在电脑上启动并登录 `Relay local bridge`
2. 该设备保持在线并连接到 Relay 云端中继
3. 用户在手机 Web 登录同一账号
4. 用户看到自己的电脑设备状态为在线
5. 用户进入该设备
6. 用户切换到目标 workspace
7. 用户通过对话发起 AI coding 任务
8. 云端将该请求转发到电脑上的 `Relay local bridge`
9. `Relay local bridge` 调用本地 `codex cli`
10. 执行结果以流式方式返回到手机端
11. 当前 session 被完整记录并同步
12. memory 数据可在多端保持一致

如果这条链路无法稳定闭环，则不能认为达到 `v0.2.0`。

---

## 6. 必须完成的功能目标

### 6.1 Local Relay Bridge

必须支持：

- 本地常驻运行
- 登录账号
- 持续维持与云端的连接
- 管理本机 workspace
- 调用本地 `codex cli`
- 将执行流转发给远端客户端

这是 `v0.2.0` 的前提能力。

没有常驻 `Relay local bridge`，就没有远程访问基础。

### 6.2 云端中继

必须支持：

- 手机端与本地 bridge 之间的连接桥接
- 会话同步
- memory 同步
- 设备在线状态管理

这个服务的产品目标不是对外展示，而是：

- 稳定
- 低延迟
- 持续连接
- 对用户透明

### 6.3 手机端远程对话

必须支持：

- 选择在线设备
- 切换 workspace
- 查看当前任务状态
- 流式收发消息
- 在手机端继续 session

界面上应优先保证：

- 输入快
- 输出连贯
- 长连接稳定
- 任务状态明确

### 6.4 数据同步

必须支持：

- session 完整内容同步
- memory 完整内容同步
- 多端登录后数据一致

同步到云端数据库的数据，至少包括：

- 账号信息
- 设备信息
- workspace 元信息
- session 内容
- memory 内容

### 6.5 设备状态与错误反馈

必须支持：

- 在线 / 离线状态展示
- 连接失败时的清晰提示
- 设备离线时的不可执行提示

最低要求：

- 用户必须明确知道问题出在：
  - 设备离线
  - 网络异常
  - 连接中断

而不是笼统地“没反应”

---

## 7. 非功能性目标

用户已经明确指出，`v0.2.0` 的第一优先级是性能与连接质量。

因此以下非功能要求是本版本的核心目标，而不是附加项。

### 7.1 低延迟

必须尽可能降低手机端远程使用时的感知延迟。

目标不是理论延迟最小，而是用户主观感受到：

- 很快
- 反馈立即开始
- 不拖泥带水

### 7.2 持续连接

必须优先保证：

- 保持连接
- 不轻易断线
- 会话不中断

如果发生网络波动，系统应优先尝试：

- 保活
- 自动恢复
- 继续当前会话

### 7.3 流式体验

必须支持优秀的流式体验：

- 返回过程连续
- 不要长时间空白后一次性刷出
- 任务执行中用户能持续获得反馈

### 7.4 手机端体验

手机端必须是“可用且舒适”的，不是桌面页面硬缩放。

必须重点优化：

- 输入区
- 阅读区
- 长消息滚动
- 长连接状态提示
- 对话节奏

### 7.5 稳定性优先于花哨功能

`v0.2.0` 的优先级排序应明确为：

1. 连接稳定
2. 流式迅速
3. 会话可继续
4. 多端同步
5. 更复杂的附加功能

---

## 8. v0.2.0 页面与产品形态目标

### 8.1 桌面端

桌面端继续保留完整工作台形态：

- workspace
- sessions
- memories
- readme
- settings

它仍是完整信息密度最高的一端。

### 8.2 手机端

手机端应做成更克制的远程工作界面。

核心页面优先级建议为：

1. 登录 / 设备选择
2. 当前设备状态页
3. 远程对话执行页
4. session 列表
5. memory 查看页

当前版本中，手机端不是“桌面端三栏布局缩小版”。

手机端重点是：

- 远程对话
- 任务状态
- 快速切换 workspace

### 8.3 设备页

建议新增一个明确的设备层入口，用于展示：

- 当前账号下的设备列表
- 在线状态
- 当前可连接电脑

这会成为 `v0.2.0` 相比 `v0.1.0` 的新增核心页面能力。

---

## 9. v0.2.0 版本边界

为了避免版本继续膨胀，以下能力不作为 `v0.2.0` 必须项：

- 手机端文件树浏览
- 手机端深度文件编辑
- 复杂记忆编辑器
- 多人协作
- 团队权限系统
- 完整设备风控后台
- 插件市场
- 高级知识图谱
- 离线设备上的任务排队执行

这些能力可以进入后续版本，但不应挤压 `v0.2.0` 的核心目标。

---

## 10. v0.2.0 完成标准

当以下条件同时满足时，可以认为 Relay 达到 `v0.2.0` 目标：

- 电脑端存在常驻 `Relay local bridge`
- `Relay local bridge` 可以稳定在线
- 手机端可在不同网络环境下访问该在线设备
- 手机端可切换 workspace
- 用户可通过对话远程驱动本地 `codex cli`
- 结果以流式方式返回到手机端
- session 完整内容可多端同步
- memory 完整内容可多端同步
- 账号支持多端登录
- 设备离线时，系统明确提示“此设备离线，无法执行任务”
- 连接与流式体验达到“稳定、迅速、不断线”的产品要求

---

## 11. 当前阶段最关键的工程问题

如果要推进 `v0.2.0`，优先级最高的不是页面视觉，而是以下工程问题：

1. `local relay bridge` 进程模型
2. 云端中继连接模型
3. 设备在线状态与会话恢复策略
4. 手机端消息流协议
5. session / memory 云端同步模型

这些问题会直接决定：

- 性能
- 稳定性
- 用户实际体验

---

## 12. 下一步建议

基于目前已明确的目标，下一步建议不是继续泛化讨论，而是开始写 `v0.2.0` 的系统架构草案。

建议顺序：

1. 写 `local relay bridge + 云端中继 + Web 前端（桌面 / 手机）` 的系统拓扑
2. 定义设备、连接、session、memory 的数据模型
3. 定义远程消息流协议
4. 定义在线 / 离线 / 重连状态机
5. 再回到页面与交互细化
