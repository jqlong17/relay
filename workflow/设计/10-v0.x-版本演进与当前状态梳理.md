# Relay v0.x 版本演进与当前状态梳理

## 1. 文档目的

本文件用于基于当前仓库中的：

- `workflow/session/*.md`
- `workflow/设计/*.md`
- `workflow/执行计划/*.md`

重新梳理 Relay 到目前为止各个版本号对应做了什么、哪些已经真正落地、哪些只是设计过但尚未完成，并给出当前阶段的统一判断。

这份文档的目标不是重写历史，而是解决三个现实问题：

1. 当前仓库里同时存在 `v0.0.x`、`v0.1.x`、`v0.2.0-alpha/beta` 等不同层级的命名，容易混淆
2. 一些版本在文档中有明确计划，但实际实现只完成了其中一部分
3. 现在已经进入“账号 + 设备 + 在线心跳”阶段，需要重新归位到一个更适合后续推进的版本体系中

---

## 2. 结论先行

如果只看“原始规划”，Relay 的版本演进大致是：

- `v0.0.1` 本地最小链路
- `v0.0.2` 个人远程可达验证
- `v0.0.3` sessions 整理台 / 时间线记忆
- `v0.1.0` 本地正式工作台
- `v0.1.1` 记忆能力插件化与双端共用
- `v0.1.2` 目标驱动自动推进内部规则
- `v0.2.0` 账号登录后连接自己本地 Codex 的远程正式版

如果看“当前实际落地”，更准确的判断是：

- `v0.0.1` 已完成并远超
- `v0.0.2` 已完成
- `v0.0.3` 核心能力已完成并并入主线
- `v0.1.0` 核心能力已大体完成
- `v0.1.1` 和 `v0.1.2` 有规划文档，但不是当前主线交付重点
- `v0.2.0` 已完成前置基础设施的一大半：
  - GitHub 登录
  - 设备绑定
  - 默认设备
  - 在线心跳
  - Render 公网壳子
- 但最关键的“云端长连接中继 + 远程执行路由”尚未完成

因此，当前阶段最适合对外和对内统一表达为：

- 当前处于：`0.1.2`
- 当前版本语义：`账号与设备在线基础设施完成版`
- 下一步目标：`0.1.3` 最小长连接中继版
- 里程碑目标：`0.2.0` 正式远程执行版

## 6. 下一步最合理的版本目标

下一步不建议再发散做新的独立功能线。

下一步应该只做一件最关键的事：

- 建立最小 `Realtime Relay`

具体而言：

1. 浏览器建立 `client websocket`
2. 本地 `local-bridge` 建立 `agent websocket`
3. 云端维护在线设备连接映射
4. `/workspace` 将 run 请求路由到默认在线设备
5. 本地执行事件再经由云端实时回流到浏览器

这一步做完，Relay 才会从：

- “能登录、能看到自己的在线设备”

升级为：

- “登录后真的能在网页端连接自己的本地 Codex”
