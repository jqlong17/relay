# Relay v0.x 单人团队产品演进路线图

## 1. 文档目的

本文件用于结合 Relay 当前代码现状、目标产品形态和单人团队现实约束，规划接下来多个版本的演进路线。

本文件重点回答以下问题：

1. Relay 当前处于哪个阶段
2. 最终目标应该拆成哪些版本逐步实现
3. 哪些能力应该先跑通，哪些能力应该后补
4. 每个版本的目标、边界、验收标准和风险是什么
5. 单人团队在账号、服务器、存储和实时架构上应如何控制复杂度

本文件不是逐任务级别的 TDD 计划，而是产品与技术联合路线图。

---

## 2. 当前阶段判断

结合当前仓库实现，Relay 当前大致处于：

- 本地工作台阶段已完成主要基础
- 个人远程访问版尚未完整闭环
- 平台化远程接入版尚未开始正式实现

更具体地说，当前系统已经具备：

- 本地 Web 工作台
- 移动端页面骨架
- 本地 `local-bridge`
- workspace / session / memory / automation 基础对象
- 运行时流式事件链路
- 与本地 Codex 的执行集成

当前系统尚未具备：

- 账号体系
- 设备模型
- 设备绑定
- 云端 API
- 云端实时中继
- 本地 Agent 主动连接云端
- 多用户隔离
- 正式公网版部署闭环

因此，Relay 当前并不是“还没成型”，而是：

- 本地单机版本已经走到中段
- 远程平台化版本还停在设计完成、实现待开始的阶段

---

## 3. 路线图设计原则

### 3.1 先跑通主链路，再补齐体系化能力

对单人团队来说，最危险的做法不是功能少，而是：

- 一上来同时做账号
- 同时做设备绑定
- 同时做实时中继
- 同时做 session 云同步
- 同时做 memory 云同步
- 同时做完整移动端产品化

这样会让项目在技术上过早进入高复杂度区。

更合理的路线是：

1. 先跑通个人远程访问闭环
2. 再建立账号与设备模型
3. 再打通云端中继主链路
4. 最后再做同步、可靠性和产品细节完善

### 3.2 单人团队优先“少系统数”

你的核心约束不是算力不足，而是维护能力有限。

因此路线图必须遵循：

- 尽量少自建基础设施
- 尽量少供应商
- 优先买成熟认证和数据库能力
- 只把精力放在产品核心链路

### 3.3 每个版本必须能自证价值

每个版本都必须回答一个明确问题，而不是做一堆散功能。

例如：

- `v0.0.2` 回答的是“人不在电脑前，能不能远程继续用”
- `v0.2.0-alpha` 回答的是“一个账号能不能绑定一台设备并稳定转发执行”
- `v0.2.0` 回答的是“这件事能不能成为正式可用产品”

---

## 4. 最终目标定义

Relay 最终目标可以定义为：

- 用户有 Relay 账号
- 用户可以在自己的电脑上安装并运行本地 Agent
- 用户可以在网页端或手机端登录该账号
- 用户可以看到自己名下设备的在线状态
- 用户可以选择设备和 workspace
- 用户可以通过对话驱动本地 Codex 执行任务
- 结果以流式方式稳定返回
- session / memory / automation 能在多端维持一致

为了达到这个目标，建议把后续版本分成六段。

---

## 5. 推荐版本路线

## 5.1 `v0.0.2` 个人远程可达版

### 核心目标

回答一个问题：

- 用户能不能在外网通过手机访问自己电脑上的 Relay，并继续调用本地 Codex

### 本版本必须完成

- Web 登录页
- 单密码访问保护
- `httpOnly` session cookie
- 未登录拦截页面访问
- 未登录拦截 `/api/bridge/*`
- Cloudflare Tunnel 接入文档
- 手机端基本可用

### 本版本明确不做

- 用户账号体系
- 多用户
- 云端数据库
- 设备绑定
- 云端实时中继

### 技术实现建议

- 继续使用现有 `apps/web + services/local-bridge`
- Web 前面增加服务端鉴权
- 只把 Web 暴露到公网
- `local-bridge` 仍只监听本地回环地址

### 验收标准

- 外网手机可以打开 Relay
- 登录后可以继续对话
- 可以看到流式返回
- 未登录用户不能访问主工作区与 bridge API

### 为什么这一版值得先做

因为它是最低成本验证：

- 远程使用场景是否真实成立
- 手机端交互是否顺手
- 用户对“远程驱动本地 Codex”有没有明确价值感

### 风险

- 不具备平台化能力
- 仍然只适合个人使用
- 不能作为正式多用户产品架构

---

## 5.2 `v0.1.3` 本地能力收敛版

### 核心目标

在进入云端之前，进一步收敛本地能力边界，让后续 Agent 化更稳。

### 本版本必须完成

- 统一 session / run / runtime 语义
- 收敛前端 API 边界
- 明确本地 `local-bridge` 中哪些模块是：
  - 本地执行核心
  - 本地状态持久化
  - UI 代理层
- 为 Agent 化预留领域模型：
  - `Device`
  - `Run`
  - `DeviceStatus`

### 本版本明确不做

- 真正的云端接入
- 真实账号登录
- WebSocket 中继

### 技术实现建议

- 在当前仓库中先把本地执行核心与 Web 代理层分离
- 把运行态建模从“页面事件”提升到“领域对象”

### 验收标准

- 当前本地功能不回归
- 运行时状态模型更清晰
- `local-bridge` 更容易拆成“执行核心 + agent 连接层”

### 为什么这一版重要

如果直接在当前结构上生硬叠加云端能力，后面会越来越难维护。

这一版的价值在于：

- 让后续平台化改造不至于破坏现有本地能力

---

## 5.3 `v0.2.0-alpha` 账号与设备接入验证版

### 核心目标

回答两个问题：

- 账号登录能不能稳定工作
- 一台本地设备能不能绑定到账号并在线显示

### 本版本必须完成

- 正式账号系统接入
- 用户登录 Web
- 创建设备绑定码
- 本地 Agent 输入绑定码完成设备绑定
- 设备在线状态展示
- 设备列表页

### 本版本建议选型

- 认证：`Supabase Auth`
- 登录方式：
  - `Google`
  - `Email Magic Link`
- 数据库：`Supabase Postgres`
- 存储：先用 `Supabase Storage`
- DNS / TLS：`Cloudflare`

### 本版本明确不做

- 完整远程执行
- session / memory 全量云同步
- 多设备复杂调度

### 技术实现建议

- 保持本地执行仍走现有链路
- 先只做账号与设备接入层
- `local-bridge` 开始演进为 `Local Agent`

### 验收标准

- 用户能登录 Web
- 用户能看到自己的设备
- 设备可以绑定、上线、下线
- 设备归属关系正确

### 为什么这一版单独拆出来

因为账号和设备接入本身就是一个独立复杂度层。

先验证这层，能避免把“登录问题”和“实时执行问题”混在一起难以排查。

---

## 5.4 `v0.2.0-beta` 远程执行主链路版

### 核心目标

回答最关键的问题：

- 网页端登录账号后，能不能稳定连接自己本地设备并流式执行

### 本版本必须完成

- Cloud API 最小版本
- Realtime Relay 最小版本
- Agent 到云端的长连接
- 浏览器到云端的长连接
- 按 `deviceId` 路由 run.start
- 流式输出转发
- 在线设备可执行
- 离线设备明确报错

### 本版本明确不做

- 复杂 memory 编辑
- 多设备之间任务迁移
- 企业级权限体系
- 完整后台运营系统

### 技术实现建议

- 常驻服务器部署：
  - `Cloud API`
  - `Realtime Relay`
- 本地 Agent 常驻连接
- 事件流用 WebSocket
- session / memory 先做最小元数据同步

### 验收标准

- 登录后能选择在线设备
- 能进入 workspace
- 能发消息
- 能看到流式返回
- 设备离线时提示明确

### 为什么这一版是里程碑

因为一旦这一版跑通，Relay 就从“本地产品”真正跨进了“远程平台产品”。

---

## 5.5 `v0.2.0` 正式可用版

### 核心目标

把 `beta` 阶段的主链路从“能跑”提升为“可正式使用”。

### 本版本必须完成

- 浏览器重连恢复
- Agent 自动重连
- 心跳与设备状态机
- run 生命周期完整化
- session 云同步
- memory 云同步基础版
- 运行记录与错误日志
- 更完善的移动端体验

### 本版本建议补齐

- 首包优化
- 流式渲染节流
- 状态提示统一
- 设备 last seen / degraded 状态
- 基础监控与告警

### 验收标准

- 正常网络下体验稳定
- 弱网下状态可解释
- 重连后不容易丢任务态
- 多端看到的 session / memory 基本一致

### 这一版的产品意义

这一版才是你可以对外说：

- Relay 已经是一个正式的远程 Agent 工作台

---

## 5.6 `v0.3.0` 产品完善与效率增强版

### 核心目标

在主链路稳定后，开始补体验和效率能力。

### 可做方向

- 多设备管理体验优化
- 更成熟的 session / memory 策略
- 文件树与附件远程能力增强
- automation 与远程执行的联动
- 更强的设备健康检查
- 更强的运行态历史与检索

### 本版本不应过早做的事

- 企业组织权限
- 团队协作复杂模型
- 大而全的后台系统

对单人团队来说，`v0.3.0` 仍然应该优先做“增强核心闭环”，而不是扩散到太多旁支。

---

## 6. 推荐技术选型路线

### 6.1 账号体系建议

不建议先自建邮箱密码系统。

推荐顺序如下：

1. `Supabase Auth`
2. 登录方式优先：
   - `Google`
   - `Email Magic Link`
3. 若用户更偏开发者，再加：
   - `GitHub`

原因如下：

- 认证本身不是 Relay 的核心壁垒
- 单人团队维护密码体系成本太高
- 社交登录与 Magic Link 更适合快速上线

### 6.2 服务器建议

推荐：

- `1 台常驻云服务器`
- 部署：
  - `Cloud API`
  - `Realtime Relay`
- `Cloudflare` 做 DNS 与 HTTPS

不建议一开始把核心实时链路建立在纯 serverless 方案上。

### 6.3 存储建议

推荐分层：

- `Supabase Postgres`
  - 用户、设备、run、session、memory 元数据
- `Supabase Storage`
  - 附件、导出文件、大对象
- 本地设备
  - 本地文件和高敏感原始上下文

### 6.4 本地端建议

推荐把当前 `services/local-bridge` 演进为：

- `local execution core`
- `agent connectivity layer`

而不是推倒重写。

---

## 7. 为什么要按这个顺序推进

### 7.1 `v0.0.2` 先做，是为了验证真实使用场景

在远程产品里，最大的未知数不是数据库表，而是：

- 用户真的会不会在外面用手机继续驱动本地 Codex
- 这个场景到底有多高频

先做 `v0.0.2`，能快速验证这个问题。

### 7.2 `v0.2.0-alpha` 单独做账号与设备，是为了隔离复杂度

如果把账号、设备、实时转发一起上，出了问题很难知道是：

- 登录不稳
- 设备绑定不稳
- 还是中继不稳

拆开做更适合单人团队调试。

### 7.3 `v0.2.0-beta` 再做远程执行，是为了保证主链路优先

对 Relay 来说，最核心价值不是“看设备列表”，而是：

- 登录后真的能远程执行自己的本地 Agent

所以账号与设备验证通过后，应立刻转到主链路打通。

### 7.4 正式版再补同步和稳定性，是为了避免过早优化

session 云同步、memory 同步、运行恢复都重要，但它们不是第一性前提。

先跑通，再稳定，再增强，这是更稳的顺序。

### `v0.0.2`

完成定义：

- 外网手机登录后可以继续使用自己电脑上的 Relay

### `v0.1.3`

完成定义：

- 本地执行核心与后续 Agent 化边界清晰

### `v0.2.0-alpha`

完成定义：

- 账号登录、设备绑定、设备在线状态全部可用

### `v0.2.0-beta`

完成定义：

- 账号登录后可远程流式执行自己本地设备上的 Codex

完成定义：

- 远程执行稳定、状态清晰、同步基础可用

### `v0.3.0`

完成定义：

- 核心闭环体验被显著增强，而不是只增加边缘功能

## 9. 单人团队的现实建议

### 9.1 不要同时启动过多战线

最容易失败的路径通常是：

- 一边重构本地 bridge
- 一边做 auth
- 一边做云 API
- 一边做移动端全量体验
- 一边做 memory 云同步

建议任何时候只保留一条主线和一条辅助线。

### 9.2 不要自己造登录系统

认证与密码管理对单人团队极不划算。

这部分应该购买成熟能力，把时间留给：

- Agent
- Device
- Realtime
- Session / Memory

### 9.3 不要过早做“看起来平台化”的边缘功能

例如：

- 团队权限
- 企业组织
- 多角色后台
- 复杂计费

这些都可以等主链路跑通后再说。

### 9.4 文档与版本边界要严格

由于你依赖 AI coding，版本边界必须写得很清楚，否则 AI 很容易在实现过程中把多个版本目标混在一起。

因此建议：

- 每个版本都有单独的产品目标文档
- 每个版本都有单独的执行计划
- 每个版本都有明确“本次不做”

---

## 10. 推荐下一步

基于当前阶段，建议你的接下来动作顺序如下：

1. 先完成 `v0.0.2`
2. 然后补一个 `v0.1.3` 的本地能力收敛版
3. 再开始 `v0.2.0-alpha` 的账号与设备接入
4. 然后推进 `v0.2.0-beta` 的远程执行主链路
5. 最后收敛到 `v0.2.0` 正式版

如果只能选一个立刻动手的版本，建议优先做：

- `v0.0.2`

因为它成本最低，且最能验证你的核心场景是否成立。

如果 `v0.0.2` 验证有效，再开始 `v0.2.0-alpha`，那时投入云端平台化会更有把握。

---

## 11. 结论

Relay 的正确路线不是“一次性做成完整云平台”，而是：

- 先让个人远程可用
- 再让账号与设备可接入
- 再让远程执行主链路成立
- 最后把稳定性和同步能力补齐

对单人团队来说，这条路线的关键不是功能总量，而是：

- 每一步都能形成闭环
- 每一步都能减少不确定性
- 每一步都不把基础设施复杂度拉得过高

因此，推荐版本顺序为：

- `v0.0.2`
- `v0.1.3`
- `v0.2.0-alpha`
- `v0.2.0-beta`
- `v0.2.0`
- `v0.3.0`

这是从当前代码现状走向最终产品目标的最稳妥路线。
