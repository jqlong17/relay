# Relay v0.1.0 产品目标

## 1. 文档目的

本文件用于明确 Relay `v0.1.0` 的产品目标、用户旅程、功能边界与非功能要求。

它的作用不是描述“最终愿景”，而是定义一个相对完整、可以真实使用、可以继续演进的首个正式版本。

`v0.1.0` 不是 `v1`。

这意味着：

- 版本必须完整可用
- 但范围必须克制
- 优先把核心链路跑通
- 不追求一次覆盖所有高级能力

---

## 2. 版本定位

Relay `v0.1.0` 的定位是：

一个以 Web 为界面的本地 Agent 工作台，用户可以打开本地文件夹作为工作区，基于对话驱动底层 `codex cli` 执行任务，沉淀 session，并将 session 整理为可编辑的长期记忆。

这不是一个“网页终端皮肤”，也不是一个“纯聊天产品”。

它的重点是把这几个对象真正串起来：

- 本地工作区
- 对话与执行
- session 沉淀
- memory 整理
- readme 文档
- settings 配置

---

## 3. 目标用户

当前默认用户是：

- 使用 AI coding / AI agent 的个人用户
- 主要在本机工作区中完成任务
- 希望自然语言优先，但也接受一定命令感交互
- 需要看到 AI 在做什么，而不只是得到一个回答
- 希望长期积累记忆，而不是每次重新开始

---

## 4. v0.1.0 核心用户旅程

`v0.1.0` 需要完整支持如下主链路：

1. 用户在左侧打开本地某个文件夹，将其作为 `workspace`
2. 用户进入该工作区，开启一个新的对话 session
3. 用户在中间主区域输入任务，底层由 `codex cli` 执行
4. 界面以流式方式返回对话与执行过程
5. 当前 session 被保存，并可在左侧 / `sessions` 页面查看
6. 用户可以在 `sessions` 页面中，通过 AI 对话的方式，将一个或多个 session 整理为 memory
7. memory 会以文档形式沉淀，并按编号保存
8. `memories` 页面以日期入口展示这些记忆
9. 用户点击某一天，可以查看该日的记忆内容
10. 用户可以编辑和修改已有记忆
11. 系统支持每 20 轮对话自动整理一次 memory 文档
12. `readme` 页面展示产品介绍与项目说明，并支持 Markdown 渲染
13. `settings` 页面支持配置项查看、编辑、保存、重置

如果这条主链路不能闭环，就不能视为达到 `v0.1.0`。

---

## 5. v0.1.0 必须完成的功能目标

### 5.1 Workspace

必须支持：

- 打开本地文件夹作为 workspace
- 展示当前 workspace 的基本信息
- 在 workspace 下创建和切换 session
- 在右侧展示本地文件树
- 中间区域展示当前 session 的主对话 / 执行流

当前版本重点：

- 先围绕单用户、本机工作区
- 不要求多端同步
- 不要求远程共享 workspace

### 5.2 对话与执行

必须支持：

- 用户通过自然语言发起任务
- 底层使用 `codex cli` 作为执行内核
- 前端接收并展示流式返回
- 对话和执行过程要在主区域中真实可见

最低要求：

- 不是一次性返回整段结果
- 必须具备流式感
- 用户能感知任务正在进行中

### 5.3 Sessions

必须支持：

- 按 workspace 浏览 session
- 查看某个 session 的多轮对话内容
- session 被真实记录，而不是仅存在内存里

`sessions` 页要承担：

- 历史回看
- 多 session 引用
- 记忆整理入口

### 5.4 Memories

必须支持：

- 用户通过 AI 对话方式，将 session 整理为 memory
- memory 可以被保存为文档
- memory 文档使用 `01`、`02` 这类顺序编号
- 每份 memory 文档都按时间线记录过程

这里有一个关键约束：

memory 不是“只有结论的摘要”。

每份 memory 文档必须记录：

- 用户提出了什么
- AI 建议了什么
- 中间如何收敛
- 最后形成了什么决策

也就是说，memory 应该更接近：

- 时间线式决策记录

而不是：

- 单纯的结论列表

### 5.5 自动整理规则

`v0.1.0` 必须支持一条默认自动化规则：

- 每 20 轮对话后，自动整理一次 memory 文档

最低要求：

- 规则真实生效
- 自动整理结果可见
- 用户能在 `memories` 页面中找到对应产物

### 5.6 Memories 页面

必须支持：

- 以日期为入口展示 memory
- 点击某一天查看该日具体记忆
- 查看记忆正文
- 编辑和修改记忆内容

当前版本重点：

- 先把“按日期浏览 + 查看 + 编辑”跑通
- 暂不追求复杂主题聚类、语义检索、图谱关系

### 5.7 Readme

必须支持：

- 展示对 Relay 产品的介绍
- 支持 Markdown 渲染
- 可作为项目说明页存在

### 5.8 Settings

必须支持：

- 展示配置内容
- 编辑配置
- 保存配置
- 重置配置

当前已明确：

- 使用 `relay.ui.toml`
- language 与 UI 配置统一放在该配置文件中

## 6. 页面目标

### 6.1 workspace

页面目标：

- 成为主工作台
- 用户在这里发起任务并观察执行

页面结构：

- 左：workspace 下的 session 列表
- 中：主对话 / 执行流
- 右：文件树

### 6.2 sessions

页面目标：

- 查看历史 session
- 基于一个或多个 session 整理 memory

页面结构：

- 左：按 workspace 分组的 session 索引
- 中：当前 session 对话详情
- 右：memory copilot 对话区

### 6.3 memories

页面目标：

- 浏览长期记忆
- 按日期查看记忆沉淀结果
- 编辑记忆

页面结构：

- 左：日期 / 日历入口
- 右：当日记忆详情

### 6.4 readme

页面目标：

- 展示产品介绍与项目说明
- 提供 Markdown 阅读体验

### 6.5 settings

页面目标：

- 提供系统配置入口
- 支持直接编辑 `relay.ui.toml`

---

## 7. 非功能性目标

`v0.1.0` 必须满足以下非功能要求。

### 7.1 流畅性

整体使用要流畅。

这不是可选项，而是版本目标的一部分。

具体包括：

- 输入不卡顿
- 切换页面不卡顿
- 对话流更新自然
- 流式返回有连续感
- 不出现明显阻塞和跳闪

### 7.2 产品体验

必须有优秀的产品使用体验。

这意味着：

- 结构清晰
- 操作路径短
- 主次分明
- 反馈明确
- 没有大量解释性噪音文案

### 7.3 渲染质量

必须有较好的渲染效果。

具体包括：

- Markdown 渲染效果优秀
- 中英文排版稳定
- 长文本、代码块、标题、列表显示清楚
- 文件树、会话流、记忆文档具备良好可读性

---

## 8. v0.1.0 版本边界

为了避免版本膨胀，以下能力不作为 `v0.1.0` 必须项：

- 多用户协作
- 云端 workspace
- 手机端完整适配与远程控制闭环
- 复杂权限系统
- 完整插件系统
- 高级记忆检索与知识图谱
- 多种自动化规则编辑器
- 完整 diff / merge 工作流
- 大而全的命令模式体系

这些方向可以是后续版本目标，但不应挤进 `v0.1.0`。

---

## 9. v0.1.0 完成标准

当以下条件同时满足时，可以认为 Relay 达到 `v0.1.0` 目标：

- 用户可以打开本地文件夹作为 workspace
- 用户可以发起对话，底层真实调用 `codex cli`
- 中间主区域可以流式展示返回过程
- session 可以被记录并回看
- 用户可以在 `sessions` 页面通过 AI 对话整理 memory
- memory 会以编号文档形式保存
- memory 文档按时间线记录过程，而不只是结论
- 系统支持每 20 轮自动整理一次 memory
- `memories` 页面可按日期查看和编辑记忆
- `readme` 页面支持 Markdown 渲染
- `settings` 页面支持配置编辑、保存、重置
- 整体交互流畅，产品体验达标

---

## 10. 下一步建议

基于这个版本目标，下一阶段最合理的工作不是继续做纯视觉，而是开始定义 `v0.1.0` 所需的数据模型与接口合同。

建议按这个顺序推进：

1. 定义 `workspace / session / runtime / memory` 类型
2. 定义前端 API contract
3. 先用 mock 数据跑通页面状态
4. 再对接 `codex cli`
5. 最后补 memory 自动整理与文档保存链路
