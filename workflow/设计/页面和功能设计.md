# Relay 页面和功能设计

## 1. 文档目的

本文件用于整理 Relay 当前已经明确的产品雏形，重点从两个角度描述：

- 页面布局
- 页面承担的功能职责

目标是让后续接手的人在不回看完整对话的情况下，也能理解当前产品结构、功能边界和后续可继续开发的方向。

---

## 2. 当前产品定位

Relay 是一个 web 端的 agent 工作台。

当前阶段的核心目标不是完整跑通所有后端能力，而是先明确：

- 用户在 web 端如何组织 workspace、session、memory
- 不同页面分别承担什么任务
- 本机 runtime、对话、文件上下文、长期记忆在界面上如何分工

当前默认用户画像：

- 使用 AI coding / AI agent 的个人用户
- 主要在本机工作区中运行任务
- 偏向自然语言工作流，但也支持更强命令感的操作方式

---

## 3. 顶层信息架构

当前顶层结构已经基本确定为：

- `workspace`
- `sessions`
- `memories`
- `readme`

另外新增了一个独立于主导航之外的系统入口：

- `settings`

说明：

- `workspace / sessions / memories / readme` 是主业务页面
- `settings` 是次级系统入口，不属于一级业务导航
- `readme` 仍然保留为主页面，不并入设置

---

## 4. 整体页面结构原则

当前产品已经明确采用工作台式布局，而不是单栏内容页。

默认布局分工：

- 左侧：索引、入口、导航、列表
- 中间：主工作流 / 主阅读流 / 主对话流
- 右侧：上下文、细节、结果、辅助面板、配置

不同页面会根据任务重点做比例调整，但原则不变。

---

## 5. workspace 页面

### 5.1 页面角色

`workspace` 是当前主工作区页面。

它代表：

- 当前正在进行中的项目工作区
- 当前选中的 session 所在环境
- 与本地文件上下文结合最紧密的页面

### 5.2 当前布局

#### 左侧

按 workspace 分组的 session 列表：

- workspace 名称
- workspace 对应 branch
- workspace 下的多个 session

特点：

- session 单行显示
- 激活 session 有细色条
- branch 只放在 workspace 层，不放在 session 层

#### 中间

当前 workspace 的主工作流区域。

当前原型已经收敛为：

- 顶部：当前 workspace 标识
- 中间：更接近对话流 / 工作日志流的消息区域
- 底部：输入框与执行按钮

它表达的是：

- 当前激活 session 的工作过程
- 用户与 agent 的当前往返
- 下一步要继续执行什么

#### 右侧

当前默认只显示文件树：

- 文件树
- 目录层级
- 当前文件高亮

原则：

- 文件属于 workspace 内部上下文
- 不作为顶层页面

### 5.3 当前承担的功能

- 在多个 workspace 中切换当前项目
- 在当前 workspace 下切换 session
- 查看当前工作流消息
- 输入新的任务或指令
- 查看当前本地文件树

### 5.4 后续可继续开发的功能

- 对接真实 session 数据
- 对接真实文件树
- 对接本机 runtime 状态
- 支持真正执行任务
- 支持命令模式 / 自然语言模式切换
- 支持文件预览与编辑上下文

---

## 6. sessions 页面

### 6.1 页面角色

`sessions` 是历史会话与记忆整理页面。

它不只是“看聊天记录”，而是负责：

- 按 workspace 浏览历史 session
- 查看某个 session 的完整内容
- 基于一个或多个 session 整理 memory

### 6.2 当前布局

#### 左侧

按 workspace 分组的 session 索引：

- workspace 名称
- branch
- 每个 workspace 下的 session 列表
- 时间信息

#### 中间

当前选中 session 的完整多轮对话内容：

- session 标题
- 所属 workspace 信息
- user / assistant 多轮消息

#### 右侧

`memory copilot`

当前已经明确只负责一件事：

- 基于当前或多个 session 进行记忆整理

当前形态：

- memory chat 消息流
- 输入区
- 支持 `@session` 引用多个 session

### 6.3 当前承担的功能

- 浏览历史 session
- 查看单个 session 详情
- 将多个 session 整理为更稳定的 memory
- 作为 `memories` 系统的输入来源

### 6.4 已明确的产品原则

- 自动规则不放在 `sessions` 右侧
- `sessions` 右侧只保留即时整理工作流
- 系统级 memory automation 归属 `memories`

### 6.5 后续可继续开发的功能

- 支持真正选择多个 session
- 支持 `@session` 自动补全
- 支持生成 memory 草稿
- 支持确认保存 memory
- 支持用户自定义 memory 提示词

---

## 7. memories 页面

### 7.1 页面角色

`memories` 是长期记忆系统页面。

它负责：

- 浏览已经沉淀的 memory
- 按时间查看每天产生的记忆
- 查看某条记忆来源于哪些 session
- 查看 memory automation 的状态

### 7.2 当前布局

#### 顶部

极简头部：

- 页面标识
- `run now`

不放冗余说明文案。

#### 左侧

时间轴 / 日历入口：

- 月份列表
- 日期方块
- 产出密度高低
- 当前选中日期
- 轻量主题筛选

定位：

- 这是入口，不是主内容

#### 右侧

当前选中日期的详情面板：

- 日期
- daily summary
- memories 列表
- source sessions
- automation

这是页面的主内容区域。

### 7.3 当前承担的功能

- 按时间查看 memory 产出
- 查看某天沉淀了哪些记忆
- 查看这些记忆来自哪些 session
- 查看自动化规则和运行状态
- 手动触发一次整理

### 7.4 当前已经明确的组织原则

- 时间是主入口
- 主题是辅助筛选，不是主导航
- 右侧详情是页面主要视觉重点
- automation 属于 memories 系统的一部分

### 7.5 后续可继续开发的功能

- 真正按日期切换数据
- 真正按主题筛选
- 查看 memory 的创建方式：auto / manual / refined
- 查看 automation 最近运行记录
- 支持编辑 memory automation 规则

## 8. readme 页面

### 8.1 页面角色

`readme` 是项目文档与上下文页面。

它代表：

- 产品说明
- 项目约束
- 交接阅读入口
- 长文档阅读位置

### 8.2 当前承担的功能

- 作为独立文档页面存在
- 保持与主工作流分离

### 8.3 后续可继续开发的方向

- 展示真实 README 内容
- 支持渲染 markdown
- 展示项目规则、运行方式、上下文文档

## 9. settings 面板

### 9.1 页面角色

`settings` 是系统级次级入口，不是主页面。

它当前放在顶栏最右侧，通过右侧轻面板展开。

### 9.2 当前布局

当前是一个轻量右侧面板。

目前只包含：

- `language`
- `中文 / English` 切换

### 9.3 当前承担的功能

- 保存界面语言偏好
- 作为后续系统设置的容器

### 9.4 后续可继续扩展的设置项

- appearance
- memory automation defaults
- runtime / connection
- integrations

---

## 10. 当前已经确定的产品关系

### 10.1 workspace 与 sessions

- `workspace` 更关注“当前正在做什么”
- `sessions` 更关注“已经发生过哪些会话，以及如何整理它们”

### 10.2 sessions 与 memories

- `sessions` 是记忆整理的输入来源
- `memories` 是整理后的长期存储与浏览系统

### 10.3 memories 与 settings

- `memories` 承载业务层 memory automation
- `settings` 承载系统层偏好设置

两者不能混淆。

---

## 11. 当前阶段已经基本确定的功能雏形

从产品功能角度看，目前已经能明确 Relay 的基础能力模型：

### A. 工作区能力

- 管理多个本地 workspace
- 在 workspace 下组织多个 session
- 在当前 workspace 中继续运行 agent 任务

### B. 会话能力

- 查看当前与历史 session
- 在 session 之间切换
- 查看完整多轮对话

### C. 记忆能力

- 从 session 中整理 memory
- 按日期浏览 memory
- 查看 memory 来源
- 管理 memory automation

### D. 系统能力

- 提供 settings 入口
- 提供语言切换

---

## 12. 后续建议的开发顺序

如果继续往下开发，建议按以下顺序推进：

1. 把 `workspace` 接成真实数据结构
2. 把 `sessions` 的多 session 引用和 memory 生成接成真实交互
3. 把 `memories` 的日期数据和 memory 数据接真实数据源
4. 把 `settings` 的语言切换接到真实 UI 文案系统
5. 再接 runtime / app-server / codex 内核

---

## 13. 结论

当前 Relay 已经不只是一个纯视觉原型，而是一个页面职责较清晰的产品雏形。

目前已经比较稳定的结论是：

- 顶层页面划分基本确定
- 各页面主任务已经明确
- `workspace -> sessions -> memories` 的产品链路已经成形
- settings 已经有独立系统入口

后续主要工作将从“继续讨论页面是什么”转向“把这些页面接成真实数据和真实能力”。
