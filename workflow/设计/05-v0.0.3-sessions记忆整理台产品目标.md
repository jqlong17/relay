# Relay v0.0.3 产品目标

## 1. 文档目的

本文件用于定义 Relay `v0.0.3` 的产品目标。

`v0.0.3` 的重点不是远程访问，不是账号体系，也不是完整 memory 系统。

这个版本只解决一个更具体的问题：

- 把顶部 `sessions` 页面从“历史会话查看页”升级为“历史会话整理台”

这里的“整理台”不是再开一个聊天入口，而是让用户能够基于已经发生过的 session，快速整理出：

- 时间线式摘要
- 用户决策记录
- 用户关注点与不关注点
- 后续可沉淀为 memory 的候选内容

---

## 2. 版本定位

Relay `v0.0.3` 的定位是：

一个围绕历史 session 进行回看、筛选、理解、整理的工作台页面。

这个版本明确区分两个页面职责：

- `workspace`
  - 负责进行中的工作
  - 负责新建 session
  - 负责继续与 Codex 对话

- `sessions`
  - 负责历史 session 浏览
  - 负责 session 内容整理
  - 负责为 memory 沉淀提供输入

因此，`v0.0.3` 的 `sessions` 页面不是“第二个聊天页”，也不是“新建 session 页”。

---

## 3. 核心目标

`v0.0.3` 必须完成以下目标：

1. 顶部 `sessions` 页面使用真实的 Codex persisted threads，而不是静态 mock 数据
2. 用户可以在 `sessions` 页面高效浏览和切换历史 session
3. 用户可以在页面中清晰阅读某个 session 的完整对话时间线
4. 右侧 `memory copilot` 明确服务于“整理 session”，而不是普通详情展示
5. 页面内提供一条默认可用的快捷整理提示词，直接产出高质量整理结果
6. 该整理结果围绕时间线、用户决策、用户关注点展开，而不是泛泛摘要

如果这 6 件事没有闭环，则不能认为达到 `v0.0.3`。

---

## 4. 核心用户旅程

`v0.0.3` 需要完整支持如下最短链路：

1. 用户进入顶部 `sessions` 页面
2. 左侧看到真实历史 session 列表
3. 用户选择一个 session
4. 中间看到该 session 的完整多轮对话内容
5. 右侧默认显示“快捷整理”入口
6. 用户点击或触发默认整理提示词
7. 系统基于当前 session 生成结构化整理结果
8. 用户据此决定是否继续 refine，或后续保存为 memory

这条链路是 `v0.0.3` 的主线。

---

## 5. 关键产品判断

### 5.1 `sessions` 页不负责新建 session

这个版本明确规定：

- `sessions` 页不提供 `new session`
- `sessions` 页不承担“开始工作”的职责
- 新建 session 的入口只保留在 `workspace`

原因：

- 避免页面职责重叠
- 避免把历史整理页做成第二个工作台
- 保持用户心智清晰：
  - `workspace` = 做事
  - `sessions` = 整理已经发生过的事

### 5.2 右侧不是详情栏，而是整理面板

`sessions` 页右侧不再定义为 metadata/detail panel。

右侧必须明确是：

- `memory copilot`

其职责不是重复展示 session 信息，而是帮助用户把 session 转换为更稳定的认知材料。

### 5.3 默认整理结果不是“结论摘要”

这个版本明确要求默认整理结果更接近：

- 时间线式整理记录

而不是：

- 一段泛化总结
- 只有结论的条目列表
- 脱离上下文的泛泛提炼

---

## 6. 页面目标

### 6.1 左侧：Session 索引

必须支持：

- 展示真实历史 session
- 清楚显示标题与更新时间
- 支持切换当前 session
- 支持后续扩展搜索 / 筛选

当前版本不要求：

- 新建 session
- 在该页直接继续运行任务

### 6.2 中间：Session 时间线

必须支持：

- 查看完整对话内容
- 保持 user / assistant 时间线可读
- 对空 thread、未 materialize thread 提供清晰反馈

中间区域的目标不是“聊天输入”，而是“可回看的对话材料”。

### 6.3 右侧：Memory Copilot

必须支持：

- 展示一个明确的默认快捷整理能力
- 用户基于当前 session 发起整理
- 后续可扩展到基于多个 session 整理

当前版本右侧优先级是：

- 先把“整理”跑通
- 暂不追求复杂自动化规则配置

---

## 7. 默认快捷整理能力

### 7.1 默认提示词

`v0.0.3` 必须内置一条默认可用的快捷整理提示词。

推荐产品文案为：

- 请按照时间线梳理 session 中的对话摘要，保留必要细节，尤其要写清楚涉及到的具体文件。请记录用户做出的决策及其理由，但理由只有在用户明确说出时才记录；如果用户没有明确说明，不要推断。核心关注用户在对话过程中究竟关注什么、不关注什么，并把这些点清楚记录下来。

这条提示词不是普通 placeholder，而是：

- 右侧默认整理能力
- 第一条最重要的快捷操作
- 后续 memory 沉淀的基础输入

### 7.2 默认输出目标

基于这条快捷提示词，系统生成的内容至少应覆盖：

- 时间线摘要
- 关键文件与改动对象
- 用户明确表达过的决策
- 用户明确表达过的决策理由
- 用户反复强调的关注点
- 用户明确排斥、弱关注或要求不要做的内容

### 7.3 默认输出约束

默认整理必须遵守以下约束：

- 必须按时间线组织，而不是打散重组
- 必须保留必要上下文，不能只写抽象结论
- 只要提到文件，就尽量写清楚具体文件路径或文件名
- 只有用户明确说过的理由，才能写成“理由”
- 若理由来自模型推测，必须省略，不能伪装成用户原意
- 对“关注点 / 不关注点”的提炼，必须尽量基于对话中的明确表达、重复强调、反复修正来总结

### 7.4 推荐输出结构

为了保证整理质量，产品层建议输出结构至少包含以下部分：

- 本次 session 做了什么
- 时间线摘要
- 涉及的关键文件
- 用户做出的明确决策
- 用户明确给出的理由
- 用户关注的内容
- 用户不关注或明确排除的内容
- 后续可沉淀为 memory 的候选点

当前版本不强制 UI 完全固定为这个结构，但默认 prompt 和默认结果必须向这个结构靠拢。

---

## 8. 数据与交互范围

### 8.1 当前版本必须接真实数据

`sessions` 页必须直接读取真实 session 数据源。

当前真实来源为：

- Codex persisted threads

这意味着：

- 左侧列表来自真实 thread 列表
- 中间详情来自真实 thread 内容
- 不能继续停留在完全静态 mock 状态

### 8.2 当前版本优先单 session 整理

`v0.0.3` 先把以下路径做稳：

- 选择一个 session
- 生成整理结果

但需要在产品结构上预留：

- 多 session 选择
- `@session` 引用
- 二次 refine
- 保存为 memory

### 8.3 当前版本不要求完成的事情

`v0.0.3` 不要求完成：

- 完整 memory 存储系统
- memory 自动化规则体系
- 多端 memory 同步
- 复杂知识库检索
- 在 `sessions` 页继续运行 Codex 任务

---

## 9. 验收标准

满足以下条件，可认为 `v0.0.3` 的 `sessions` 页面达到目标：

1. 页面展示真实 session 列表，而不是静态样例
2. 点击任一 session 后可查看真实对话详情
3. 页面中不存在 `new session` 主入口
4. 右侧明确呈现“整理 session”的交互定位
5. 页面提供默认快捷整理提示词
6. 该默认整理目标明确要求：
   - 按时间线梳理
   - 保留文件细节
   - 只记录用户明确给出的理由
   - 提炼用户关注点与不关注点

---

## 10. 后续版本自然延伸

在 `v0.0.3` 完成后，后续版本可以自然继续推进：

- 多 session 联合整理
- 整理结果保存为 memory 草稿
- 人工 refine memory
- memory 文档正式保存
- 基于规则自动触发整理

但这些都不应该反过来稀释 `v0.0.3` 的核心目标：

- 先把 `sessions` 页面做成一个真正有用的历史整理台
