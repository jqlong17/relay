# Relay v0.0.2 产品目标

## 1. 文档目的

本文件用于定义 Relay `v0.0.2` 的产品目标。

`v0.0.2` 不是完整的 `v0.2.0` 远程产品，也不是多端同步版本。

它的目标只有一个：

- 以最快可交付的方式，让用户在不同网络下，通过手机访问并使用自己电脑上的 Relay / Codex 能力

这个版本优先解决：

- 电脑在公司或家里
- 手机在外面
- 手机浏览器可以打开 Relay
- 用户可以通过对话继续驱动本地 Codex 执行任务

本版本明确采用：

- `Cloudflare Tunnel`

并增加：

- 一个强密码访问层

---

## 2. 版本定位

Relay `v0.0.2` 的定位是：

一个基于本地 `Relay local-bridge + Relay web`，通过 `Cloudflare Tunnel` 暴露到公网、并用强密码保护访问的远程 Web Agent 原型。

用户不需要：

- 配置公网 IP
- 配置路由器端口映射
- 和电脑处于同一局域网

用户只需要：

1. 在电脑上启动 Relay
2. 启动 Cloudflare Tunnel
3. 在手机浏览器打开公网地址
4. 输入访问密码
5. 继续和本地 Codex 对话

这个版本不追求账号体系，不追求云数据库，不追求完整 session / memory 同步。

它只回答一个问题：

- Relay 能不能尽快从“本机 Web”升级为“异地可访问的个人 Agent 工作台”

---

## 3. 核心目标

`v0.0.2` 必须完成以下目标：

1. 本地电脑上的 Relay Web 可以通过 Cloudflare Tunnel 暴露到公网
2. 手机在不同网络环境下可以访问这个 Relay 地址
3. 访问前必须经过强密码认证
4. 认证通过后，用户可以继续现有 workspace / session，对本地 Codex 发消息
5. 前端仍然保持流式返回
6. 当本地 bridge 或 tunnel 不可用时，页面有清晰错误提示

如果这 6 件事不能闭环，则不能认为达到 `v0.0.2`。

---

## 4. 核心用户旅程

`v0.0.2` 需要完整支持如下最短链路：

1. 用户在公司电脑启动 Relay
2. 用户启动 Cloudflare Tunnel，把 Relay Web 暴露到公网
3. 用户在手机浏览器打开对应公网地址
4. 用户首先看到密码验证页
5. 输入正确密码后进入 Relay 工作区
6. 用户选择现有 workspace / session
7. 用户发送一条自然语言任务
8. Relay 通过本地 `local-bridge` 调用本地 Codex
9. 手机端流式看到返回内容
10. 用户刷新页面后，只要密码会话仍有效，仍可继续使用

这条链路是 `v0.0.2` 的唯一主线。

---

## 5. 架构边界

`v0.0.2` 不引入完整云端后端。

采用的实际架构是：

- `Relay Web`
- `Relay local-bridge`
- `Cloudflare Tunnel`

其中：

- `Relay Web`
  - 继续运行在本地电脑
  - 通过 `/api/bridge/*` 与本地 bridge 通信

- `Relay local-bridge`
  - 继续运行在本地电脑
  - 调用本地 Codex app-server / Codex CLI

- `Cloudflare Tunnel`
  - 只负责把本地 `web` 服务安全暴露到公网
  - 不承担业务逻辑

这意味着本版本：

- 不做云端 session 持久化
- 不做云端 memory 同步
- 不做账号登录
- 不做多设备管理

---

## 6. 必须完成的功能目标

### 6.1 远程访问

必须支持：

- 使用 Cloudflare Tunnel 暴露 `apps/web`
- 用户通过公网 URL 访问 Relay
- 不要求用户自己做端口映射或公网网络配置

当前版本建议：

- tunnel 先作为开发 / 个人使用配置，不做企业级接入方案

### 6.2 强密码认证

必须支持：

- 进入主工作区前必须先通过密码验证
- 密码配置在本地环境或配置文件中，不硬编码进前端
- 登录成功后生成一个本地签名会话
- 未登录用户不能访问：
  - workspace 页面
  - bridge API

当前版本重点：

- 一个强密码
- 一个本地 session cookie
- 一个明确的 logout 能力

当前版本不要求：

- 用户名密码体系
- OAuth
- 多用户
- 密码找回

### 6.3 手机端远程对话

必须支持：

- 在手机浏览器打开 Relay
- 查看现有 session
- 切换 session
- 发送消息
- 流式看到 AI 返回

当前版本不要求优先支持：

- 复杂文件树操作
- 深度文件编辑
- memory 编辑
- 完整桌面布局等比复刻

### 6.4 错误与状态反馈

必须支持：

- bridge 不在线时的提示
- tunnel 未配置或访问失败时的提示
- 密码错误时的明确提示
- 登录失效时自动回到登录页

### 6.5 部署与启动

必须支持：

- 在本地电脑一键启动 Relay 开发服务
- 明确文档说明如何启动 Cloudflare Tunnel
- 明确文档说明密码如何配置

当前版本优先：

- 手工部署可跑通

当前版本不要求：

- 自动化安装向导
- 桌面菜单栏应用

---

## 7. 页面目标

### 7.1 Login

新增一个最小登录页。

页面目标：

- 输入访问密码
- 提交后写入本地认证 cookie
- 成功后进入 workspace
- 失败时给出错误提示

### 7.2 Workspace

继续作为核心主页面。

在 `v0.0.2` 中的要求：

- 手机端可用
- 对话优先
- session 可切换
- 流式返回清楚

### 7.3 Sessions

可以继续保留，但不是远程版本的主验收项。

当前版本只要求：

- 在手机端至少可读、可切换

### 7.4 Settings

需要新增最小远程配置说明：

- 当前 tunnel 模式说明
- 当前密码是否已配置
- logout

---

## 8. 非功能目标

### 8.1 安全

虽然 `v0.0.2` 是快速版本，但不能裸奔公网。

最低要求：

- 未认证用户无法进入主工作区
- 未认证用户无法调用 `/api/bridge/*`
- 密码校验在服务端完成
- cookie 必须为 `httpOnly`

### 8.2 可用性

必须能重复演示。

至少保证：

- 本地启动稳定
- 隧道接入稳定
- 登录稳定
- 对话稳定

### 8.3 流式体验

手机端必须依然能看到明显的流式过程，而不是只显示最终结果。

### 8.4 低复杂度

这个版本的重点是尽快可达，不引入过早复杂度。

因此明确不做：

- 云数据库
- 多账号
- 多设备归属
- 远程文件系统写入控制面
- session / memory 云同步

---

## 9. 明确不做的内容

以下内容不属于 `v0.0.2` 必须项：

- 完整 `v0.2.0` 的账号体系
- 云端设备管理
- session 云同步
- memory 云同步
- 多端一致性保证
- 企业级访问控制
- 设备在线状态云端看板
- push 通知

---

## 10. 版本判断标准

如果用户能够在手机上，通过公网地址和密码访问自己电脑上的 Relay，并继续与本地 Codex 对话、看到流式返回，那么 `v0.0.2` 就达标。

如果仍然只能在同一网络或本机访问，那么这个版本就没有完成目标。
