## 目标

这套工具不是为了替代真机测试，而是为了把以下链路一次性记录完整：

- 事件有没有发生
- `visualViewport` 有没有变化
- CSS 变量有没有写到真正的布局宿主
- 消费节点的 computed style 有没有变化
- 最终 `rect` 有没有真的移动

## 如何开启

任选其一：

- URL 参数：`/mobile?debugMobileLayout=1`
- 控制台或书签脚本：

```js
localStorage.setItem("relay.debug.mobile-layout", "1");
location.reload();
```

关闭：

```js
localStorage.removeItem("relay.debug.mobile-layout");
sessionStorage.removeItem("relay.debug.mobile-layout.log.v1");
location.reload();
```

## 当前会记录什么

- `pointer.down`
- `touch.start`
- `focus.in`
- `focus.out`
- `viewport.update`
- `layout.sync`
- `composer.focus`
- `composer.blur`
- `composer.focus.stabilize`

每条记录都包含：

- `viewport`
- `scroll`
- `vars`
- `computed`
- `rects`
- `focus.activeElement`

## 面板能力

开启后，移动端页面会出现浮动诊断面板，支持：

- 查看最近 20 条事件
- 复制全部 JSON
- 清空当前会话日志

日志同时会保存在 `sessionStorage`：

- `relay.debug.mobile-layout.log.v1`

所以用户真机复现一次后，即使页面轻微重绘，只要同一个 tab 还在，日志仍可导出。

## 适用场景

- iOS Safari / 微信内置浏览器输入框首次点击异常
- 键盘弹出后 fixed / sticky 元素位置不对
- safe area、topbar、composer、thread 留白错位
- “JS 明明 setProperty 了，但 UI 没变化”的问题

## 排查顺序

固定按下面顺序看，不要跳：

1. 先看事件：点击、touch、focus 是否发生
2. 再看 viewport：`innerHeight`、`visualViewport.height`、`offsetTop`
3. 再看变量：是否写到 `main.mobile-app`
4. 再看 computed：`composerTop` 是否真的变了
5. 最后看 rect：输入框位置是否真的移动

## 这套工具要守住的原则

- 记录“我写了什么”不够，必须同时记录“最终节点读到了什么”
- 调试日志默认关闭，只有显式开启才出现
- 事件日志只打关键断点，不做无意义噪音采样
- E2E 用来防回退，真机诊断仍然是定位 iOS 键盘问题的主手段
